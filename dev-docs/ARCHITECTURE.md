# Gamer Rater - Technical Architecture

## System Overview

Gamer Rater is a full-stack web application with a Django REST Framework backend and a React frontend. The backend exposes a token-authenticated REST API that the React client consumes. The two sides are completely decoupled -- the backend serves only JSON, never HTML pages.

The backend handles all business logic, data persistence, authentication, and permission enforcement. The frontend is responsible for rendering, user interaction, and making HTTP requests to the API.

## Entry Points

### Server startup

The debugger in VS Code (`.vscode/launch.json`) runs `manage.py runserver`. The server starts at `http://127.0.0.1:8000`.

### Request flow

1. Request arrives at `raterproject/urls.py`
2. The DRF `DefaultRouter` matches the URL to the registered ViewSet and action
3. The ViewSet method runs (list, retrieve, create, update, partial_update, destroy)
4. The method uses a `ModelSerializer` to validate input or serialize output
5. Django's ORM handles the database read or write
6. A `Response` object is returned with JSON data and an HTTP status code

### Manual routes (outside the router)

Three endpoints are wired manually in `urls.py` because they are custom actions, not standard CRUD:

- `POST /login` -- `UserViewSet.user_login`
- `POST /register` -- `UserViewSet.register_account`
- `GET /me` -- `UserViewSet.me`

## Component Breakdown

### Authentication layer

DRF's `TokenAuthentication` is the default for all endpoints. Tokens are issued on login and registration. Every request (except login and register) must include:

```
Authorization: Token <token>
```

The `UserViewSet.get_permissions()` method exempts `register` and `login` from authentication. All other endpoints require a valid token via `permission_classes = [permissions.IsAuthenticated]`.

### ViewSets

Each resource has its own ViewSet in `raterapi/views/`. All ViewSets extend `viewsets.ViewSet` (not `ModelViewSet`) -- this means every action is written explicitly rather than inherited. This keeps each method readable and easy to customize.

Each ViewSet that writes data handles two things the serializer cannot do on its own:

- Assigning `request.user` to the model field (e.g. `serializer.save(user=request.user)`)
- Catching `IntegrityError` from unique constraint violations and returning a 400

### Serializers

Each ViewSet has a co-located `ModelSerializer`. Key patterns used across the codebase:

- `is_owner`: a `SerializerMethodField` that compares `request.user` to the object's owner field. Requires `context={"request": request}` to be passed when instantiating the serializer.
- Nested serializers: `Game` embeds `CategorySerializer(many=True, read_only=True)` to return full category objects in responses. The write side (assigning categories) is handled manually in the ViewSet via `game.categories.set(category_ids)`.
- `game_id` vs `game`: ForeignKey fields must use the model field name (`game`, not `game_id`) in the serializer fields list for DRF to correctly map incoming data.

### Permission model

| Action | Who can do it |
| --- | --- |
| Register / Login | Anyone |
| GET any resource | Authenticated users |
| POST / PUT / PATCH / DELETE games, ratings, pictures | Authenticated users |
| POST / PUT / DELETE categories | Admin users (`is_staff=True`) only |

Admin status is granted only through the Django admin panel. The registration endpoint always sets `is_staff=False` regardless of request data.

### Admin panel

Available at `http://127.0.0.1:8000/admin/`. All six models are registered. Requires a superuser account created via `python manage.py createsuperuser`.

## Data Model

Full schema is defined in `raterapi/docs/ERD.dbml`.

Summary of relationships:

- `Player` has a one-to-one link to Django's built-in `User`. Adds a `bio` field.
- `Game` has a ForeignKey to `User` (the user who added it) and a ManyToMany to `Category` through `GameCategory`.
- `GamePicture` has ForeignKeys to both `Game` and `User`.
- `GameRating` has ForeignKeys to both `Game` and `User` (stored as `player`). Unique together on `(game, player)` -- one rating per user per game.
- `GameCategory` is the explicit through model for the `Game` <-> `Category` ManyToMany.

## API Design

Base URL: `http://127.0.0.1:8000`

The DRF router generates standard REST routes for each registered resource:

| Method | URL | Description |
| --- | --- | --- |
| POST | `/register` | Create a new user account |
| POST | `/login` | Authenticate and receive a token |
| GET | `/me` | Return the authenticated user's profile |
| GET / POST | `/games` | List all games / create a game |
| GET / PUT / PATCH / DELETE | `/games/<id>` | Retrieve, update, or delete a game |
| GET / POST | `/categories` | List / create (admin only) |
| GET / PUT / DELETE | `/categories/<id>` | Retrieve / update / delete (admin only for writes) |
| GET / POST | `/gameratings` | List / create ratings |
| GET / PUT / PATCH / DELETE | `/gameratings/<id>` | Retrieve / update / delete a rating |
| GET / POST | `/gamepictures` | List / create picture URLs |
| GET / DELETE | `/gamepictures/<id>` | Retrieve / delete a picture |

### Query parameter filtering

- `GET /gameratings?game=<id>` -- filter ratings by game
- `GET /gamepictures?game_id=<id>` -- filter pictures by game

## Infrastructure and Deployment

Currently development-only:

- **Database:** SQLite (`db.sqlite3`), excluded from git
- **Environment:** managed via Pipenv, virtualenv stored outside the project directory
- **Dev server:** Django's built-in `runserver` via VS Code debugger
- **CORS:** configured in `settings.py` to allow requests from `localhost:3000` and `localhost:5173`

No production deployment is configured yet.

## Key Technical Decisions

- **`viewsets.ViewSet` over `ModelViewSet`:** keeps each action explicit and easy to customize, at the cost of more boilerplate
- **Token auth over session auth:** stateless, works cleanly with a decoupled React frontend
- **`unique_together` on GameRating:** enforced at the database level, not just application logic
- **`is_staff` for admin gating:** reuses Django's built-in field rather than adding a custom role model
- **Categories write-protected at the view level:** each write method checks `request.user.is_staff` and returns 403 if false

## Project Conventions

### Code style

- One model per file in `raterapi/models/`
- One ViewSet and its serializer per file in `raterapi/views/`
- All models exported from `raterapi/models/__init__.py`
- All ViewSets exported from `raterapi/views/__init__.py`
- Line length: 120 characters (configured in `.pylintrc` and `pylint.args`)

### Error handling

- `DoesNotExist` exceptions are caught in retrieve, update, and destroy methods and return 404
- `IntegrityError` from unique constraint violations is caught in create methods and returns 400 with a descriptive message
- Permission failures return 403 (not 404) so the client knows the resource exists but is forbidden

### Password rules

Passwords are validated against Django's built-in validators plus a custom `StrongPasswordValidator` that requires at least one uppercase letter, one lowercase letter, one digit, and one symbol.

### Database seeding

`seed_database.sh` wipes the database, regenerates migrations, and loads fixtures in dependency order: `users` → `categories` → `games` → `gameratings` → `gamecategories`. After seeding, user accounts must be created via `POST /register` since user passwords cannot be stored as plain text in fixtures.
